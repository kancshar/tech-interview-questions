## Core Fundamentals

### 1. What is the difference between concurrency and parallelism, and when would you use each in a backend service? 🟢

**Answer:** Concurrency is about handling multiple tasks by making progress on them over time, while parallelism is executing multiple tasks at the same time on multiple cores or machines. In backend services, concurrency is useful for I/O-bound workloads like API calls, while parallelism helps with CPU-bound tasks such as heavy data processing.

*Difficulty: Easy*

---

### 2. How would you investigate a web service that becomes slow only during peak traffic hours? 🟡

**Answer:** I would start by checking latency, error rates, CPU, memory, and database metrics to identify the bottleneck. Then I would review logs and traces to see whether the issue is caused by application code, database contention, external dependencies, or resource exhaustion under load.

*Difficulty: Medium*

---

### 3. Why is idempotency important in distributed systems, and how would you implement it for an API endpoint? 🟡

**Answer:** Idempotency prevents duplicate side effects when clients retry requests due to timeouts or network failures. A common approach is to require an idempotency key, store the result of the first successful request, and return the same result for repeated requests with the same key.

*Difficulty: Medium*

---

### 4. A service depends on a third-party API that is occasionally slow or unavailable. What architectural patterns would you use to make your system resilient? 🔴

**Answer:** I would use timeouts, retries with backoff, circuit breakers, and fallback behavior to prevent cascading failures. For non-critical operations, I might also use queues or asynchronous processing so the core user flow remains responsive even when the dependency is degraded.

*Difficulty: Hard*

## Python

### 1. What is the difference between a list and a generator in Python, and when would you prefer a generator? 🟢

**Answer:** A list stores all items in memory, while a generator produces items lazily as they are needed. Generators are preferable for large datasets or streaming data because they reduce memory usage and can improve performance for sequential processing.

*Difficulty: Easy*

---

### 2. How does Python's GIL affect multithreading, and what approach would you take for CPU-bound versus I/O-bound tasks? 🟡

**Answer:** The GIL allows only one thread to execute Python bytecode at a time, which limits true parallelism for CPU-bound work in standard CPython. For I/O-bound tasks, threads or async can still work well, while CPU-bound tasks are better handled with multiprocessing or external workers.

*Difficulty: Medium*

---

### 3. How would you structure error handling and logging in a Python service so that production issues are easier to debug? 🟡

**Answer:** I would catch exceptions at clear boundaries, log structured context such as request IDs and key inputs, and avoid swallowing errors silently. I would also separate expected business errors from unexpected system failures and ensure logs are consistent enough to support tracing and alerting.

*Difficulty: Medium*

---

### 4. You need to optimize a Python function that processes millions of records and is currently too slow. What steps would you take before rewriting it? 🔴

**Answer:** I would first profile the code to identify the actual bottleneck instead of guessing, then reduce unnecessary allocations, repeated computations, and expensive Python-level loops where possible. Depending on the findings, I might use built-in functions, batching, vectorized libraries, caching, or move hot paths to multiprocessing or a compiled extension.

*Difficulty: Hard*

## LangChain

### 1. What problem does LangChain solve in LLM applications, and when is it useful versus calling an LLM API directly? 🟢

**Answer:** LangChain helps organize common LLM application patterns such as prompt composition, retrieval, tool use, memory, and chaining multiple steps together. It is useful when the application has orchestration complexity, but for simple single-prompt use cases a direct API call is often simpler and easier to maintain.

*Difficulty: Easy*

---

### 2. How would you design a basic retrieval-augmented generation pipeline for internal company documents? 🟡

**Answer:** I would split documents into chunks, generate embeddings, store them in a vector database, and retrieve the most relevant chunks for each user query. Then I would pass the retrieved context to the LLM with a prompt that constrains the answer to the provided sources and ideally returns citations.

*Difficulty: Medium*

---

### 3. A LangChain-based chatbot gives inconsistent answers even when relevant documents exist. How would you debug it? 🟡

**Answer:** I would inspect each stage separately: chunking quality, embedding model choice, retrieval results, prompt instructions, and the final model output. In practice, many issues come from poor chunk boundaries, weak retrieval settings, or prompts that do not clearly force the model to use the retrieved context.

*Difficulty: Medium*

---

### 4. What are the main trade-offs when adding memory and tool-calling to a LangChain agent in production? 🔴

**Answer:** Memory can improve continuity but also increases token usage, latency, and the risk of carrying forward irrelevant or sensitive information. Tool-calling makes the agent more capable, but it adds orchestration complexity, failure modes, and the need for strong guardrails, observability, and permission controls.

*Difficulty: Hard*

## System Design & Architecture

### 1. How would you containerize a Python application with Docker for reliable deployment across environments? 🟢

**Answer:** I would use a minimal base image, pin dependencies, copy only the required files, and define a clear entrypoint for the service. I would also use a .dockerignore file, run as a non-root user when possible, and keep configuration external through environment variables or secrets management.

*Difficulty: Easy*

---

### 2. Design a service that accepts document uploads, processes them asynchronously, and exposes status to users. What components would you include? 🟡

**Answer:** I would use an API service for uploads, object storage for files, a queue for background jobs, worker processes for document processing, and a database to track job state. The client can poll a status endpoint or receive updates through webhooks or websockets depending on the product needs.

*Difficulty: Medium*

---

### 3. How would you design observability for a microservice-based application running in containers? 🟡

**Answer:** I would include centralized logs, metrics, and distributed tracing so issues can be correlated across services. Key practices include structured logging, request correlation IDs, dashboards for latency and error rates, and alerts tied to service-level objectives rather than only infrastructure metrics.

*Difficulty: Medium*

---

### 4. You need to design a scalable LLM-powered API that serves many users while controlling cost and latency. What architectural decisions would you make? 🔴

**Answer:** I would separate synchronous and asynchronous workloads, cache repeated results where safe, use rate limiting, and choose model tiers based on request complexity. I would also add retrieval to reduce prompt size, monitor token usage closely, and design fallbacks so the system can degrade gracefully under load or provider issues.

*Difficulty: Hard*

## Problem Solving & Algorithms

### 1. If you need to check whether two large lists have any common elements, what approach would you use and why? 🟢

**Answer:** I would usually convert one list to a set and then scan the other list, because set membership checks are typically O(1) on average. This reduces the overall time compared with comparing every pair of elements.

*Difficulty: Easy*

---

### 2. How would you detect and prevent duplicate processing of jobs in a distributed worker system? 🟡

**Answer:** I would assign each job a unique identifier and store processing state in a durable system so workers can check whether the job has already been handled. To prevent duplicates, I would combine idempotent processing with leases, deduplication keys, or transactional updates depending on the queue and storage model.

*Difficulty: Medium*

---

### 3. You have a stream of events and need to keep track of the top 100 most frequent items efficiently. What data structures would you use? 🟡

**Answer:** I would maintain a hash map for counts and use a heap or similar structure to track the current top items efficiently. If the stream is extremely large or unbounded, I might use approximate algorithms such as Count-Min Sketch to reduce memory usage.

*Difficulty: Medium*

---

### 4. How would you approach finding performance issues in an algorithm that works correctly but times out on production-sized input? 🔴

**Answer:** I would analyze the algorithm's time and space complexity, then profile it with realistic input sizes to confirm where the time is actually spent. After that, I would look for opportunities to reduce nested loops, avoid repeated work with caching or preprocessing, and choose data structures that improve the dominant operations.

*Difficulty: Hard*
